Brief alignment

  • Organized by project, not invoice document — each row is a project; expanding it reveals its draws. A GC thinks in draws, not invoice docs.
  • Billing position is the language — Contract → Billed → Paid → Outstanding → Remaining-to-bill. Tabular numerals, demoted cents. No margin or cost surfaced.
  • The cockpit is actionable — four clickable filters. Ready to bill is the rust-accent tile because that’s the one the operator should click first. Overdue renders the number in danger red without taking the rust accent. No GST-collected tile — that lives on Expenses.
  • Status badges are soft-pair tokens, not a rainbow — Draft / Sent / Paid / Overdue / Void come from status-tokens.ts. Overdue carries an inline 28d age divider. Rollup chips reuse the same soft-pair vocabulary (Ready / 1 overdue / Paid in full / Draft).
  • Henry shows up twice, quietly — once as the ready-to-bill prompt on Donahue (“Framing complete — bill draw 2?” with a single rust Bill draw button), and once as a passive trace on the overdue Hillcrest draw 4 (“Henry sent a reminder 3d ago”). Not a chat box.
  • Sort order honors the brief — ready-to-bill surfaces first, paid-in-full sinks to the bottom: Donahue → Hillcrest (expanded) → Bandsma (draft) → Petrov (paid).
  • No “New invoice” button — invoices are generated from a project’s draw schedule. Only ghost Import with Henry in the header.
  • Three type sizes on screen — 16px (project names, KPI labels-of-the-day), 14px (body / cells / customer line), 12px (status badges, sub-labels, cents, dates). Mono used only as labelling device (column headers, invoice hashes, KPI eyebrows, draw sub-labels).
Out of scope this pass: the empty state (Henry receipt icon + “Complete a job and generate an invoice to start getting paid.”) is described in the brief but not rendered here. Holdback handling, GST remittance, and the customer-facing invoice document are explicitly out of scope per the must-nots.
