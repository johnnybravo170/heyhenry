> **⚠ The boundary that governs everything (state it on the screen):** **QBO is the downstream system of record. HeyHenry is import-only today.** Every byte flows **QBO → HeyHenry** (`qbo_sync_log.direction='pull'`). The schema is *ready* for push (`direction='push'`, per-row `qbo_sync_status`, `qbo_sync_token`, the per-tenant `qbo_payment_method_map`/`qbo_tax_code_map`), and the connect card already promises *"Coming soon: every new invoice and payment auto-syncs to QBO"* — but **no push worker or push UI exists.** The design must not imply two-way sync is live. Frame it honestly: *"QuickBooks is your books. HeyHenry reads from it today; pushing invoices & costs out is coming."*
>
> **⚠ Naming correction (code wins over the request).** The class-mapping route maps **QBO Class → HeyHenry *Project*** (a QBO bookkeeper's job-cost tag pointed at a HeyHenry project so vendor spend rolls up under the right job). It does **not** map "HeyHenry categories → QBO classes" — neither the direction nor the entity matches. This brief specs what's built (Class → Project). If a future category↔class map for *push* is wanted, that's a separate target surface, flagged in Open questions. (Per `[[feedback_no_bucket_terminology]]`: this surface says **"category"** never "bucket" if/when categories enter it; today it's projects.)

**Object / workflow / role(s):** primary = the **QBO Connection** (`tenants.qbo_*`) and its **sync ledger** (`qbo_import_jobs` runs + `qbo_sync_log` attempts + the per-row `qbo_sync_status` on every synced entity); workflow = **Onboarding / Books Bridge** — *connect once, backfill history, keep HeyHenry and QBO agreeing.* Roles: **owner / admin** (operate the integration); **owner** for the connection itself (it's billing-adjacent, like Stripe); **member / worker / client** never (dashboard-only, owner/admin-gated — pairs with the `settings.md` nav-role-filter fix). **Primary action:** *connect QuickBooks, pull your history in cleanly, and resolve the handful of things Henry couldn't auto-decide — without double-entering anything.*

## Naming & boundary discipline (carry through the whole design)
- **"QuickBooks / QBO"** = the external system of record (Intuit). **"HeyHenry"** = the operating layer reading from it.
- **"Import" = pull = QBO → HeyHenry.** The only live direction. Say "import," not "sync," for anything that exists today. Reserve "sync" / "push" for the labelled **Coming-soon** affordance.
- **"Class" is a QBO term** — surface it as *"QuickBooks Class"* (the bookkeeper's job-cost tag), mapped **to a HeyHenry project.** Never silently rename it "category."
- **"Client" not "homeowner"** anywhere customer-facing language appears (per `[[feedback_client_not_homeowner]]`); but the imported data entity stays **"customer"** (the data-model term, and what QBO calls it). The review queue resolves *customers*; copy about who they are to the GC reads "client."
- **One accent: rust.** `✦` appears **only** on real Henry touchpoints (the dedup match-confidence reasoning, the class-mapping suggestion) — labelled, operator confirms. No decorative sparkles.

## Current vs target (the delta this brief drives)
The integration is **functionally complete for import** but reads as **five disconnected utility pages with raw status colors and no connection-health cockpit.** Six gaps:

1. **No status-token discipline — raw colors everywhere.** The connected banner is hand-rolled `emerald-50/200/800`; the review nudge + skipped-FK warnings are raw `amber-300/50/800`; rollback uses `destructive`; the import-launcher's per-entity skipped/failed counts use inline `amber-600`/`text-destructive`. **None** route through `status-tokens.ts` (confirmed: zero `status-tokens` importers under `features/settings`). **Target:** every sync state renders through the canonical tones — define a **`qboSyncStatusTone`** map (`synced→success`, `pending→info`, `failed→danger`, `disabled→neutral`) plus the connection/run states below, so QBO pills match invoice/project pills app-wide (PATTERNS §7). Never colour-only — carry the tone glyph.
2. **No connection cockpit.** The hub opens to a green "Connected to {company} · realm {id}" strip and three undifferentiated link cards. A GC mid-onboarding can't answer *"is my history in? is anything stuck? what needs me?"* at a glance. **Target:** a **connection-health header** (connected company · last import · environment) + a **"Needs you" line** that surfaces the review-queue count and any failed run — the one number that should pull the eye.
3. **The four pages don't know about each other.** `qbo-review` and `qbo-history` are orphan routes reachable only by deep-link; the hub's three cards are static (no counts); the import-launcher's review nudge hard-links to `/settings/qbo-review` but the reverse path is a prose nudge. **Target:** the hub's sub-route cards carry **live counts** (review: N pending · history: last run + status · class-map: N unmapped classes), and each sub-route shares the connection header + a back-to-hub crumb (the current `← Settings` crumb skips the hub).
4. **Import lives buried inside the connected card.** The highest-value action (pull my history) is a `Dialog` nested in `quickbooks-connect-card.tsx`, with a polling progress panel below it. It works, but it's the hero job and reads as a footnote. **Target:** promote import to the hub's primary action when connected; keep the deferred/resumable job model (cron-backed) and its live counters, restyled.
5. **`qbo_sync_log` + per-row `qbo_sync_status` are invisible.** The schema records every pull attempt (and is push-ready) with actor routing, and stamps `synced|pending|failed|disabled` on every entity row — **but nothing renders it.** History shows *import-job* counters, not the per-entity sync ledger. **Target (mostly future, scoped honestly):** a **sync log** view (the `qbo_sync_log` audit) and per-row sync badges become meaningful *when push lands*; today, expose the **failed-attempt** rows from `qbo_sync_log` in History so a stuck import is legible, and reserve the per-row badge surface for the push epic. Don't build a push UI here.
6. **Off-brand chrome + native-dialog parity.** Rollback + disconnect already use shadcn `AlertDialog` (good — keep), but the destructive copy and the raw-amber warnings need the Paper restyle; the class-mapping "Overwrite existing" is a bare checkbox that silently re-tags manually-classified rows. **Target:** Paper restyle throughout; make "Overwrite existing" a labelled, consequence-clear control (it's the one foot-gun on the class-map page).

**Target:** a single **QuickBooks hub** that opens on connection health + what-needs-you, promotes import to the primary action, gives the three sub-routes live counts and a shared header, renders every state through `status-tokens`, surfaces the `qbo_sync_log` failures honestly, and **states the import-only / QBO-system-of-record boundary plainly** — all restyled to Paper. Flagged inline where target ≠ current.

## The data truth this screen must reflect
- **Connection (`tenants`):** `qbo_realm_id` (the company id; **NULL = disconnected**, kept on disconnect so reconnect to the same file is clean), `qbo_company_name`, `qbo_connected_at`, `qbo_disconnected_at`, `qbo_environment ∈ sandbox|production` (sandbox must be visibly flagged — it's not real books), `qbo_last_full_sync_at`. **Tokens (`qbo_access_token`/`qbo_refresh_token`) are service-role-only — never render them; the realm id is the only safe identifier to show.**
- **Import job (`qbo_import_jobs`):** `status ∈ queued|running|completed|failed|cancelled`, per-entity `entity_counters {fetched, imported, skipped, failed}`, `api_calls_used` (shown so the operator/Mike can watch the Intuit billing meter), `review_queue[]` (ambiguous customers), `batch_ids` (per-entity → `import_batches` for granular rollback), resume cursors (`current_entity`, `entity_cursors`) for the cron that picks up a job that hit its time budget. Entities, in FK order: **Customer · Vendor · Item · Invoice · Estimate · Payment · Bill · Purchase(→ "Expenses")**.
- **Per-row sync (`qbo_sync_status` on customers/invoices/quotes/`project_costs`/payments/bills):** `synced|pending|failed|disabled` + `qbo_*_id` + `qbo_sync_token`. **Import-only today** — these get stamped by future push; the design reserves their badge surface but doesn't fake it.
- **Sync ledger (`qbo_sync_log`):** one row per attempt — `direction ∈ pull|push`, `entity_type`, `status ∈ queued|running|succeeded|failed`, `actor_kind ∈ gc|bookkeeper|system` + `actor_email` (routes failure email per `QBO_PLAN.md` §4.3), `import_job_id`, request/response bodies, `attempt`. **Service-role writes only.** Surface the *failed pull* rows; everything else is future/audit.
- **Class → Project map (`project_costs` + `bills`, denormed `qbo_class_id|name`):** `listClassMappingsAction` rolls up **distinct QBO Class names across bills + `project_costs` (source_type='receipt')**, sorted by total spend, with the current top project per class. **Note the divergence:** the UI copy says "bills and expenses" but the query reads `project_costs` (legacy `expenses` table dropped, migration `…042909`). Fix the copy to "bills and receipts" (or "costs") on restyle. `applyClassMappingAction` bulk-updates `project_id`; `preserveExisting` (the "Overwrite existing" toggle, inverted) protects manual edits.
- **Review queue (`qbo_import_jobs.review_queue[]`):** each entry = a QBO customer that **fuzzy-matched** a HeyHenry contact (tiers `name+city` / `name`); per entry the operator picks **Merge into {candidate}** (binds `qbo_customer_id`), **Create new** (uses QBO data), or **Skip** (no DB write). Strong matches (email/phone) auto-merge upstream and never reach the queue. Emptying the queue unblocks dependents — invoices/estimates/payments/bills skipped on the first pass (parent customer unresolved) chain in on a re-run.
- **Gotcha — re-running is safe.** Every entity is de-duped on its QBO id (unique partial indexes per tenant). The import dialog says so; keep that reassurance — it's what makes "resolve review queue, then re-run" a safe loop.

## Workflow (the state machine the hub serves)
```
Disconnected ──Connect QuickBooks──▶ /api/qbo/start (HMAC state cookie) ──▶ Intuit consent
   Intuit ──approve──▶ /api/qbo/callback (verify state · exchange code · save tokens · fetch CompanyInfo)
          └─cancel / expired / error──▶ /settings?qbo=denied|invalid|error  (toast, stays disconnected)
   Connected ──Import from QuickBooks──▶ job queued ▶ running (live counters, cancellable)
          ├─ time budget hit ──▶ queued (cron resumes; "continuing in background")
          ├─ completed, clean ──▶ "Imported N records"
          ├─ completed, ambiguous customers ──▶ Review queue (N need review)
          ├─ completed, FK-skipped ──▶ "M skipped — resolve review queue, then re-run"
          └─ failed ──▶ error_message + the failed qbo_sync_log row
   Review ──merge / create / skip each──▶ queue empties ──▶ re-run chains dependents
   Any cost with a QBO Class ──▶ Class-map ──pick project · Apply──▶ spend rolls up under the job
   History ──rollback a run──▶ every record that run inserted is soft-deleted (no auto re-import)
   Connected ──Disconnect──▶ revoke at Intuit (best-effort) · clear tokens · keep realm id
```
- **Decision points:** sandbox vs production (flag it); merge vs create vs skip per ambiguous customer; overwrite-existing vs preserve on class-map; rollback (destructive, no auto-redo).
- **Failure modes the UI must handle:** OAuth denied/expired/error (4 distinct `?qbo=` returns → the existing toast set; keep, but also reflect on the disconnected hub, not only a transient toast); import hit time budget (→ "continuing in background," not "failed"); FK-skipped records (→ the explicit "resolve review, re-run" nudge); a failed run (→ surface `error_message` + the `qbo_sync_log` failure, not a dead end); accidental rollback (→ `AlertDialog` with the full consequence list — keep); silent overwrite on class-map (→ make the toggle's consequence explicit).
- **Approval / external:** the only outbound is the Intuit OAuth handshake (human-in-the-loop by definition) and the future failure-routing email (`actor_email`); **no customer-facing send originates here.**
- **Success signal:** the GC's historical books are in HeyHenry, every customer landed on exactly one record, vendor spend rolls up under the right project, and the API-call meter stayed sane.

## Henry intelligence touchpoints (labelled `✦`, operator confirms — never a chat box; per `[[henry-intelligence-not-chat]]`)
QBO import is mostly deterministic plumbing, so Henry's role is narrow and honest — **don't manufacture sparkles.** Two real touchpoints + one credit:
1. **Dedup match confidence (review queue).** The fuzzy-match tiering (`name+city` / `name`, after the deterministic email/phone auto-merge) is Henry deciding *"this QBO customer probably is your existing contact."* Surface it as a labelled **`✦` confidence chip** on each candidate — the operator still picks Merge/Create/Skip. Turns a bare tier label into a legible, undoable judgement.
2. **Class → project suggestion (class-map, target/optional).** Henry pre-stages the project `Select` for a class whose name maps cleanly to a project — `✦ suggested`, confirmed on Apply, never auto-tagged.
3. **Import-as-Henry-assisted (credit, not invention).** The pre-flight dialog + completion summary may credit Henry for the dedup/classification work it already does (*"✦ Henry matched 118 of 120 customers automatically — 2 need your eye"*). Stays fully reviewable; no new behaviour.
- **No push automations here.** When the push epic lands, *that's* where Henry-drafted invoice/cost mapping belongs — out of scope for this brief.

## Role variations
- **Owner:** full — connect/disconnect (it owns the Intuit consent + token lifecycle), import, review, class-map, rollback. The connection itself is owner-grade (billing-adjacent, like Stripe + account-delete).
- **Admin:** operate the integration — import, review, class-map, history/rollback — **but treat connect/disconnect as owner-only** (don't let an admin sever the books connection; flag for the Ops role-matrix confirm alongside `settings.md` decision #1).
- **Member:** **none.** Operational reads elsewhere don't extend to the books bridge. The nav must hide it (the `settings.md` role-filter gap); a member hitting the URL directly gets a clean "owner/admin only" state (defense-in-depth at the server).
- **Worker (`/w`) / Client (portal):** **never.** This is a dashboard back-office surface; nothing here is field- or client-facing. The boundary is absolute — no QBO data, no books, no sync state ever reaches the worker app or the client portal.

## Financial / Canadian
- **CAD throughout** — class-map spend totals render via `Money` (the schema defaults `bills.currency='CAD'`). No raw cents.
- **Tax is QBO's job here.** Imported invoices/bills **freeze** their `tax_cents` at the historical rate (PATTERNS §16 frozen-math contract) — never recompute. The per-tenant `qbo_tax_code_map` (per-province QBO TaxCode ids) is *push* config — reserved, not surfaced today.
- **No Interac/Stripe parity surface here** — that's the invoicing/payment screens. The `payments.method` enum (`cash|cheque|e-transfer|stripe|credit_card|other`) maps to QBO PaymentMethodRef via `qbo_payment_method_map` on *push* (future).
- **No holdback, no T-slips here.** T5018/year-end live in the out-of-scope bookkeeper portal (`/bk`); the `actor_kind='bookkeeper'` in `qbo_sync_log` is the only nod to that surface — failure-email routing, not a UI here.

## States
- **Disconnected (empty):** connection header in its disconnected form — Intuit Connect CTA + value prop; the three sub-route cards are **hidden** until connected (current behaviour — keep); if `qbo_disconnected_at` set, the quiet reconnect line. §6 empty-state shape (icon + headline + line + the Connect CTA).
- **Connecting / OAuth return:** the four `?qbo=` returns (`connected` success · `denied` cancelled · `invalid` expired link · `error`) → the existing toast set (keep), **plus** reflect a hard `error`/`invalid` on the disconnected header (a one-line "Last attempt didn't complete — try again"), not only a vanishing toast.
- **Connected, no import yet:** header + "Import from QuickBooks" primary + empty sub-route cards (review 0 · history "No imports yet" · class-map "No QBO Classes in your imports"). Each sub-route has its own honest empty (already built — restyle).
- **Importing (loading):** the live job panel — per-entity counters streaming, API meter, Cancel; **skeleton** on sub-route tables while their `list*` actions resolve.
- **Connected, needs-review:** the "Needs you" line leads with the review count (`warning`); the review card badge lights; the import panel's "N need review → Resolve now" strip.
- **Sync-error / run-failed:** the "Needs you" line shows "last import failed" (`danger`); History surfaces `error_message` + the `qbo_sync_log` failure row; the run pill is `danger`. **Not a dead end** — the safe-to-re-run reassurance still applies.
- **Error (action-level):** every action failure via `toast` (PATTERNS §5 `{ ok, error }`) — never native `alert()`.

## Accessibility
WCAG 2.2 AA. **Sync state never colour-only** — every `status-tokens` tone carries its glyph (connected/synced = success+check, pending = info+send, failed = danger+x, sandbox/needs-review = warning+hourglass, disabled = neutral); the sandbox flag is a labelled pill, not a color. The realm id + tokens: render only the realm id (mono), never tokens — a security *and* a noise concern. Sub-route cards are real links; merge/create/skip/apply/rollback are real buttons, keyboard-operable, ≥44px on mobile. The destructive `AlertDialog`s bring focus-trap + ESC for free (PATTERNS §3) — keep, don't hand-roll. The "Overwrite existing" toggle needs a real `<label>` + consequence text (not a bare checkbox). All run/import timestamps render in tenant tz (PATTERNS §23 — the current components use bare `new Date(...).toLocaleString()`; **fix on restyle** to `formatDate(iso, { timezone })`). Mobile grids carry `min-w-0` so class names + spend totals don't overflow (PATTERNS §18).
