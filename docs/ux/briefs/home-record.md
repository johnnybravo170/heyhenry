## Purpose
Most contractors hand off a job with a shoebox of receipts and a verbal "the paint's a Benjamin Moore something." The Home Record is the **closeout payoff** and a named market differentiator (`af12ea51`): every phase, every behind-the-wall photo, every paint code / fixture / SKU, every warranty, every approved change order — frozen into **one permanent link** (plus a branded PDF and a durable ZIP) the client keeps for repairs, insurance, resale, or the next reno. It's the last thing the client experiences from the company, so it has to *feel* like the company. Two jobs:
1. **Operator (closing the job):** assemble → **preview exactly what the client will see** → email it. Low-frequency, high-stakes, must feel finished — and must never send a half-built package or leak internal data.
2. **Client / spouse / realtor / insurer / future contractor (reading it):** open a link that works forever, on a phone or a laptop, that reads as a clean, trustworthy document from *their contractor* — and grab the PDF/ZIP to tuck away offline.

## The data truth this screen must reflect
- **The snapshot is frozen + denormalized + client-safe by construction.** `generateHomeRecordAction` copies live data at generation time; the public page only ever reads the snapshot (never live tables). The assembly queries **already enforce the client boundary**: photos require `client_visible = true` AND (`portal_tags ≠ {}` OR a `phase_id`); documents require `client_visible = true`; decisions only `status='decided'`; change orders only `status='approved'`. **Pending/internal/declined items never enter the snapshot** — the boundary is in the data layer, but the UI must never *imply* otherwise (no "0 internal photos hidden" counters that hint at hidden content).
- **The slug is permanent; only the rendering is dynamic.** Regenerating upserts in place — **same slug, fresher data** — so a link already shared keeps working. Signed URLs live ~1 week, so the public route **re-signs on every request**. This is the core promise: *"save this link — it works forever."*
- **Regeneration nulls PDF + ZIP.** Changing the snapshot sets `pdf_path`/`zip_path` to null server-side, so the derived files must be rebuilt. **The UI must make this consequence visible** (today it's silent — the buttons just flip back to "Generate") — see the readiness model (§Layout 2).
- **Three delivery formats, three durability levels.** (a) **Web link** — always available once the snapshot exists, always current, re-signed. (b) **PDF** — branded, photos *embedded as bytes* (survives source-photo deletion), but its document *links* are signed URLs that expire (~1 wk) → "a dated snapshot, regenerate to refresh links." (c) **ZIP** — **durably permanent** (actual file copies + README), the "tuck it away offline forever" artifact; also the slowest to build.
- **Photos are auto-curated, not dumped.** `curatePhotos` caps each portal-tag bucket to **12** by `ai_showcase_score` (desc, taken_at tie-break) and **always keeps phase-pinned photos** (operator-intentional). A 400-photo job becomes ≤ ~72 portal photos + all phase pins, ordered chronologically so the story reads in sequence. **This is a Henry touchpoint already shipped** (cluster #3) — credit it honestly (§Henry).
- **Behind-the-wall is a first-class section.** The public page renders `behind_wall`-tagged photos as their own section with the line *"Useful for future repairs and resale."* This is a signature value-prop of the artifact — keep it prominent, not buried in a generic gallery.
- **Frozen tenant tz.** The snapshot carries `timezone` at generation time; dates render in *that* tz (not the business's current location) so a permanent record reflects *when the work happened*. Render all dates via this (PATTERNS §23) — never bare `toLocale*`.
- **Money is light + Canadian.** The only money on the artifact is **change-order `cost_impact_cents`** (rendered CAD via `Intl … 'en-CA'`, tabular-nums). No subtotal/GST/margin math on the artifact, and **never** `unit_cost`/`markup`/supplier cost (Role Matrix boundary). GST surfaces only if a future CO line shows it — keep CAD + honest labels if so.

## Workflow (the state machine the operator flow serves)
```
No record ──Generate──▶ Snapshot exists (web link live; PDF/ZIP not built)
   Snapshot ──Generate PDF / Generate ZIP──▶ Formats building… ──▶ Ready (link + PDF + ZIP)
   Ready ──Preview as client──▶ (review the artifact) ──▶ Send
   Ready ──Email──▶ Sent (emailed_at / emailed_to stamped)  ──(Resend re-stamps)
   any   ──Regenerate (data changed)──▶ Snapshot refreshed, SAME slug, PDF+ZIP nulled ──▶ rebuild formats
```
- **Decision points:** generate now vs wait for more photos/selections; which formats to build before sending; **preview before send** (the trust gate); send to the on-file email vs an override; regenerate (and accept the rebuild) vs leave as-is.
- **Failure modes the UI must handle:** emailing with PDF/ZIP silently un-built (→ readiness model + the email checklist already greys missing formats — keep + strengthen); regenerating and forgetting to rebuild the PDF/ZIP before resending (→ a "snapshot changed since last build — PDF/ZIP are stale" warning); no client email on file (→ inline capture in the send step, already handled — keep); a ZIP timeout on a huge project (→ honest long-running state, not a frozen spinner); the operator fearing they'll leak internal photos (→ preview *is* the proof — "this is exactly what your client sees").
- **Approval / external:** the only outbound is the **email to the client** — explicit, operator-triggered, human-in-the-loop, **never automatic**. Henry may *draft* the closeout summary but **never sends** and never flips anything client-visible.
- **Success signal:** the client opens a branded link that works forever, the GC didn't fumble which files were attached, and nothing internal leaked.
