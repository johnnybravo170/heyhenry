## Positioning guardrail (carry through the whole design — non-negotiable)
This is the screen most likely to leak a SaaS seat-counting mental model, and HeyHenry's whole pitch is the opposite. **Hard rules:**
- **No per-seat pricing language. Anywhere.** No "X of N seats used," no "add a seat," no "upgrade to add users," no seat meter, no per-user cost, no "you've reached your plan limit." Adding people is **"add a guy to the crew,"** not "buy a seat." Capacity is a **footnote, not the pitch** (`[[feedback_heyhenry_intent_led_positioning]]`). The primary CTA is **"Invite a worker" / "Add to crew,"** framed as roster-building, never as a metered purchase.
- **"client" not "homeowner"** in any copy that surfaces here (the bookkeeper card already correctly says "no customer details"). Keep "customer" only where it's a fixed data-model term.
- **"category" never "bucket"** (`[[feedback_no_bucket_terminology]]`) — relevant if expense-category capability copy appears.
- **Voice is standard, not a tier-gate** — if any "what Henry can do for this worker" copy appears, never present voice as a paid add-on (`[[feedback_henry_voice_standard_all_tiers]]`).
- Tone: **Linear, not Buildertrend** — calm cream + a single rust accent, not a loud admin console.

## Purpose
The operator's **people surface**: see who's on the team, invite new crew and collaborators, set what each can do, and (for workers) set the **pay/charge rate pair** that drives Labour costing and billing. Three jobs:
1. **Add a person fast** (the hot path) — generate an invite link or fire an email; for a worker, optionally pre-load their type/capabilities/rates so they're job-ready the moment they accept.
2. **Keep the roster honest** (the long middle) — adjust a worker's capabilities or rate, change someone's role *(target)*, see which invites are still outstanding.
3. **Off-board cleanly** (rare, consequential) — a worker leaves; revoke their access without nuking history *(target: deactivate, not hard-delete)*.

It is **not** a billing/plan screen (that's Settings ▸ Billing, owner-only), **not** the worker's own profile editor (that's `/w/profile`), and **not** crew dispatch (that's `/calendar`). It's **who's on the team and what they're allowed to do.**

## The data truth this screen must reflect
- **A member is one `tenant_members` row** with a `role`. Email comes from `auth.users` (joined server-side). A worker-role member additionally has a **1:1 `worker_profiles`** row (auto-created on first list if missing).
- **Five roles, but the UI only mints three:** owner (the account holder, exactly one, **cannot be removed**), admin, member, worker, bookkeeper. Invites can only carry `worker | member | bookkeeper`; the UI today only offers **worker** and **bookkeeper**. `admin` exists in the data but has **no creation path** — flag.
- **The rate split is sensitive cost data.** `worker_profiles.default_hourly_rate_cents` = what you **pay** the worker; `default_charge_rate_cents` = what you **bill the client** for their time. The delta is your margin on labour. Owner/admin write it server-side today; **target: the charge/pay pair is owner + admin, but the *margin framing* and any "what you make on labour" cue is owner-only.** Never expose either rate to the worker themselves, to members, or anywhere client-facing.
- **Capabilities cascade:** `worker_profiles.can_log_expenses` / `can_invoice` are **tri-state** — `null` inherits the tenant default (`tenants.workers_can_*`), `true`/`false` override. The UI models this as **Default / Allow / Block**. The tenant defaults live in `WorkerDefaultsCard` at the top of the same page.
- **`worker_type` (employee | subcontractor)** changes the mental model: subs typically submit invoices (the default checkbox literally says "typical for subcontractors") and carry a `business_name`/`gst_number`; employees are paid hours. The role badge for a worker shows the **worker_type**, not the literal word "worker" (`team-members-table.tsx`).
- **The owner/admin's own labour rate** is `tenant_members.default_hourly_rate_cents` (mig `0082`) — separate from `worker_profiles`, set elsewhere (operator profile). Note it exists so the design doesn't imply only workers have rates; don't duplicate its editor here.
- **Invite lifecycle:** Active → Used / Expired (7-day) / Revoked. `used` invites can't be deleted (safety in `deleteInvite`). An emailed invite still returns a copyable link (email is best-effort, non-fatal).
- **Multi-tenant:** a single auth user can belong to several tenants (`is_active_for_user`, mig `0142`). Removing a member here removes them from **this** tenant; the hard-delete-the-auth-user behaviour is a current-state foot-gun for multi-tenant users — see Open questions.

## Current vs target (the delta this brief drives)
The screen works but reads as a **stack of four cards a developer assembled**, and it carries a seat-era ergonomic gap (no role management) plus a destructive off-board. Six gaps:
1. **Sensitive rates sit open in the list.** Every worker row **auto-expands** into a full pay/charge/capability editor (`bg-muted/20` strip, always visible). The margin-revealing **charge vs pay** pair is on-screen for every worker the moment the page loads — over-exposed, and noisy. *Target: collapse per-worker detail behind a click; gate the rate fields (esp. the charge/margin read) to owner/admin and keep them out of the default glance.*
2. **No role editor.** You can't promote a member to admin, demote an admin, or fix a mis-set role without removing and re-inviting. *Target: an inline role editor on each member row (owner-only for owner-affecting changes).*
3. **No admin invite path.** `admin` is a real role with zero UI to create it. *Target: a single "Invite" entry point that picks the role (Worker · Admin · Member · Bookkeeper), replacing the two separate cards.*
4. **Off-boarding is a hard delete.** "Remove" destroys the auth user. *Target: **deactivate** (revoke access, keep history) as the default destructive action; true delete is a deeper, rarer confirm.*
5. **Three stacked cards + two invite cards = visual sprawl.** Defaults card, members table, worker-invite card, bookkeeper-invite card, invites table — five blocks, no hierarchy. *Target: a **two-region** layout — People (members + a single invite affordance) and a quieter Defaults/settings region — restyled to Paper.*
6. **Off-brand confirm copy + no Henry.** Native-feeling `AlertDialog`s with generic copy; zero embedded intelligence on a screen that has obvious leverage (draft the invite, summarize who's idle, flag a sub missing a GST number). *Target: Paper-styled confirms with consequence-clear copy; light, labelled Henry touchpoints (below).*

**Target in one line:** a calm two-region people surface — a **roster** with inline role + capability + (owner-gated) rate editing behind progressive disclosure, a **single role-aware "Add to crew" invite flow**, **deactivate-not-delete** off-boarding, and a quiet **crew defaults** panel — all on the Paper palette, with no seat-counting anywhere. Flagged inline where target ≠ current.

## Layout *(compose from `card`, `table`, `badge`, `dialog`, `dropdown-menu`, `select`, `input`, `popover`, `alert-dialog` — invent nothing)*
The pane sits in the settings shell (`settings.md`): back-link "Settings", `h1` **Team**, sub "Invite workers and manage your team." `max-w-3xl`, single column. **Two regions** (target — today it's five stacked blocks):

1. **Region A — Crew (the people).** The hero. A header row: **"Crew"** + member count as a plain count (e.g. "4 people" — *a count, never "4 of N seats"*) + the **primary action**: a single **`Add to crew`** button (rust-accented, `Plus` icon) that opens the invite dialog (replaces the two separate invite cards — target). Below it, the **member roster** as a `table` (desktop) / stacked `card` rows (mobile):
   - **Columns:** Person (display_name or email; email as sub-line) · Role (`Badge`) · for workers, a compact capability/rate summary chip · Joined (tenant-tz) · row actions (`⋯` dropdown).
   - **Role cell:** the `Badge` becomes an **inline role editor** (target) — click → `select`/`dropdown-menu` of allowed roles. Owner row's badge is **non-editable + non-removable** (lock affordance, tooltip "The account owner can't be changed here"). A worker's badge shows **worker_type** (Employee/Subcontractor), consistent with current code.
   - **Per-worker disclosure (target — replaces the always-open strip):** a worker row expands (chevron / "Edit") into the **`worker-settings-row` editor** — Type · Log expenses (Default/Allow/Block) · Submit invoices (Default/Allow/Block) · **Pay $/hr** · **Charge $/hr** · Save. **Collapsed by default**; the rate pair is **not** in the collapsed summary for admin-without-owner if we gate margin (see Roles). Keep the existing `{ok,error}` + toast save.
   - **Row `⋯` actions:** Edit details (expand) · Change role *(target)* · Resend invite (if applicable) · **Deactivate** *(target; today: Remove → hard delete)*.
2. **Region B — Crew defaults (the quieter half).** The existing `WorkerDefaultsCard` — four checkboxes (workers can log expenses · submit invoices · edit entries >48h · auto-assign to new projects), each with its helper line. Framed as **"Defaults for everyone — override per person above."** Visually lighter than Region A (it's rare-touch config). Keep the inline helper text; it's good.
3. **Pending invites** — folds into Region A as a **secondary list** beneath the roster (or a "Pending (n)" disclosure), not its own equal-weight card. Each invite row: invitee (name / email / `code…` fallback) · **status `Badge`** (Active/Used/Expired/Revoked) · Created (tenant-tz) · actions (Copy link · Resend · Delete — Used can't delete). Empty → folded away, not an empty table.

> **Single-accent discipline:** rust is the **one** accent — spend it on the primary **`Add to crew`** CTA and active/selected nav state only. Role badges, status badges, capability chips all draw from `status-tokens` tones (neutral/info/success/etc.), **not** rust. Pay/charge figures render in the app's `money` style, muted — not coloured, not alarming.

## Progressive disclosure
- **Snapshot:** the roster — who's on the crew, their role, at a glance. Counts as a plain number. No rates, no capability detail in the default view.
- **Operational:** the invite dialog (add someone) and the inline role editor (fix a role) — the two highest-frequency edits, one click in.
- **Detail:** a worker row expanded → the capability + rate editor. **This is where pay/charge live — behind a deliberate click, owner/admin only.**
- **Audit:** invite-created and member-removed already write to `audit_log` (see `team.ts`); surface "View in audit log" as a deep-link to Settings ▸ Audit rather than rebuilding a history pane here.

## Henry intelligence touchpoints *(labelled, undoable, NOT a chat box — `[[feedback_henry_intelligence_not_chat]]`)*
This screen is config-heavy, so Henry stays **light and assistive** — no autopilot that moves people around:
- **Draft the invite email** *(target, light):* when an email is entered, a "Henry can personalize this" affordance offers a one-line tailored invite message (still sends the standard transactional template; Henry only drafts the human line). Labelled, editable, skippable.
- **Roster gaps / "needs attention" chips** *(target):* Henry flags a **subcontractor missing a GST number** (`worker_profiles.gst_number` empty — matters for T5018/year-end in `/bk`), a **worker with no pay rate set** (will cost Labour at $0), or an **invite about to expire**. Small inline chips on the relevant row, each with the fix one click away. Read-only signals, not actions.
- **Idle-crew summary** *(target, optional):* a one-line "3 of your crew haven't logged time in 14 days" surfaced quietly — informational, links to Labour, never a nudge to remove anyone.
- Frame all of the above as **"Henry noticed,"** consistent with Henry-as-intelligence; nothing here auto-edits a role, rate, or membership.

## Role variations
- **Owner:** the full surface — invite (all roles), edit roles (incl. promoting an admin), edit capabilities, **see + edit the pay/charge pair and any margin framing**, deactivate/remove. Owner's own row is self-protected (can't self-remove — enforced in `removeTeamMemberAction`).
- **Admin:** everything operational — invite (worker/member/bookkeeper; **inviting/creating another admin is owner-only — target decision**), edit capabilities, deactivate workers/members. **Rate pair: editable by admin today** (`updateWorkerCapabilitiesAction` allows admin); **target seam — keep pay/charge editable for admin but reserve any explicit *margin/"what you make"* read for owner.** Confirm with Ops (Open Q).
- **Member:** never reaches this page — `requireRole` redirects to `/dashboard`. Pairs with settings.md's nav-filtering fix (hide Team from members so the nav stops advertising it). A member hitting `/settings/team` directly should get a clean redirect/"owner & admin only" state, not a crash.
- **Worker / bookkeeper:** never — dashboard-only surface. (They *land* from here via `/join/<code>` onto `/w` or `/bk`, but never see this screen.)
- **Client / homeowner portal:** never — there is no people surface in the portal, and **no rate, margin, or other-worker data may ever cross the portal boundary.**

## Mobile vs desktop
- **Desktop:** the two-region layout; roster as a real `table` with inline role editing and row-expand. Settings is desktop "thinking work."
- **Mobile:** Team is one of the **most-used** settings on the go (an owner adds a guy from the truck). Keep it thumb-friendly: roster rows become stacked `card`s (Person + role badge + `⋯`); the **`Add to crew`** CTA is full-width and ≥44px; the invite dialog is a single-column sheet; the per-worker rate/capability editor stacks vertically (it already uses a responsive grid that collapses to 2-col — verify ≥44px row targets). Capture-first: name + email + "send" should be reachable in two taps. The deep integration-y bits (defaults) can stay below the fold.
- **No offline requirement** — this is an authenticated dashboard config surface, not a field-capture screen.

## Financial / Canadian
- **Rates are CAD**, rendered in the app `money` style. Pay/charge are **per-hour** dollar figures (stored as cents). No GST on a rate itself.
- **`worker_profiles.gst_number` + `business_name`** matter for **subcontractors** — they feed **T5018 / year-end** reporting in the bookkeeper portal (`/bk`). Surface a subtle "GST # on file" indicator for subs (and Henry's "missing GST #" chip above) so the operator isn't surprised at year-end. *(T5018 itself is out of scope — lives in `/bk`.)*
- **No holdback, no per-seat billing, no subscription anything on this screen** — plan/billing is Settings ▸ Billing (owner-only). If a teammate-count ever interacts with the plan, that lives there and is framed per `[[feedback_pricing_grandfather_principle]]`, **never** as a limit on this page.

## States
- **Empty (no crew yet — just the owner):** the roster shows the owner alone; the hero is an invite prompt — `Users` icon + "Build your crew" + "Invite workers, admins, or your bookkeeper — they'll get a link to join." + the **`Add to crew`** CTA. (PATTERNS.md §6.) **Never** "0 of N seats."
- **Empty (no pending invites):** the pending-invites list folds away entirely (no empty table) — current code shows "No invites yet"; target hides the section until there's one.
- **Loading:** per-pane `skeleton` rows for the roster (the settings sidebar stays mounted — no full reload). `listTeamMembers` does an N+1 `auth.users` lookup per member, so a skeleton matters on larger crews.
- **Error:** all mutations use the `{ ok, error }` shape + `sonner` toast (keep). Invite-email failure is **non-fatal** — show the copyable link with a quiet "couldn't send the email — copy the link instead," not a hard error.
- **Permission:** member/worker hitting the URL → clean redirect to `/dashboard` (current behaviour via `requireRole`); design the redirect, don't show a half-rendered page.
- **Owner-protected:** owner row's remove/role controls render **disabled with a tooltip** ("The account owner can't be removed or changed here"), mirroring the current `RemoveMemberButton isOwner` branch.

## Accessibility *(WCAG 2.2 AA)*
- **Roster is a real `<table>`** with `<th>` headers (or a labelled list on mobile); role/status badges are **not colour-only** — each carries text + a `status-tokens` icon (the tokens file already ships `statusToneIcon` for exactly this).
- **Inline role editor** is a real `<select>`/labelled `dropdown-menu` reachable by keyboard; the change is announced (toast + `aria-live`), not just a colour shift on the badge.
- **All targets ≥44px on mobile** — the `Add to crew` CTA, every `⋯` action trigger, the row-expand control, and the rate-editor inputs (the current grid collapses to 2-col on mobile; verify the `Save` button and selects clear 44px).
- **Destructive confirms** (deactivate / delete) state the consequence in words before the action, have a clearly non-default-styled confirm button, and trap focus (the existing `AlertDialog` does this — keep).
- **Form fields labelled** (the invite + rate inputs already use `<Label htmlFor>`); the sensitive **pay/charge** fields get explicit labels ("Pay rate — what you pay them", "Charge rate — what you bill the client") so they're never confused. Disabled owner controls expose the reason via `title`/`aria-disabled`, not just visual dimming.
- **Focus order** follows the visual order: hero CTA → roster rows → pending invites → defaults.
